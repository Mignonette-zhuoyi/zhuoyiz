01  ·  问题背景

协议在邮件线程里消失，然后要求从头再来

UW 每年处理的协议体量庞大：6,700+ 教职员工、23,000 名工作人员，FY2025 研究经费总额 $1.74 亿美元。CoMotion 与 OSP、CoE、SoM 组建了跨职能分流小组，也有一份草稿分流表单——但问题的根源是整个系统依赖个别员工脑子里的隐性经验，而不是任何可以被传递的文档。

"The messy part is when an agreement is not getting to the right unit. An agreement that belongs at CoMotion has been sent to OSP and nobody can figure out where it should belong."

— OSP 利益相关者访谈

研究者不知道从哪里开始，工作人员不确定谁负责什么。核心问题不是单一步骤的失效，而是整个系统结构性地不透明。

03  ·  研究洞察

四个结构性障碍，三条设计方向

主题分析跨越所有访谈记录与邮件案例，识别出四个相互关联的障碍，解释了为什么用户普遍体验到"慢、不透明、混乱"。

A
分流标准模糊

协议类型判断没有统一规则。"什么算研究、什么算服务"这条核心边界完全依赖人工判断——只有有经验的工作人员才知道"忽略外部标签看实质"。

B
流转过程不透明

用户提交后进入"黑盒"：协议在哪里、为什么被转手、下一步需要什么——一概不知。工作人员自己有时也无法解释路由决策，因为那是被前任做出的，没有记录。

C
机构知识碎片化

处理一份协议所需的信息分散在 PI、行政助理、法务、财务等角色手中。有经验的用户在自建私人路由图，但这无法被共享或传承。

D
复杂情境无法处理

重叠的协议历史、多资助方结构（如 HHMI 联合资助）、外部合作方的续约假设——这些情境超出现有流程的预设范围，只能靠个人经验硬撑。

1
在入口处结构化分类

不应期待用户自己判断协议类型。系统应在提交入口通过引导式问题完成初步路由，将分类从用户负担转变为系统支持。

2
让路由逻辑可见

不只告诉用户去哪里，还要解释为什么。路由判断标准、转手原因、当前状态，都需要被显式呈现，而不是隐藏在邮件线程里。

3
集中化机构知识

现在的知识活在人脑里，人走了知识就消失了。产品需要提供一个能沉淀路由逻辑、协议记录和联系人信息的持久记录层。

05  ·  Admin Portal — Workflow Builder

把隐性经验变成可编辑的 Canvas

Workflow Builder 是整个产品的设计核心。它的命题是：AI 给出可编辑的第一步，人来完成细化和校正。一个空白画布让人无从下手，完全由 AI 决定的流程图没有人会信任——Workflow Builder 设计的是这两者之间的协作界面。

📋

Definition Node

每个 Workflow 有且仅有一个，描述协议类型的基本定义与适用范围，是整棵决策树的入口。

◇

Decision Node

包含判断问题与多条件分支，是路由决策的实际执行单元，支持内联编辑问题文本和分支选项。

→

Action Node

指定研究者在每个路由结果下需要执行的具体步骤，对应实际操作指引。Approval Chain 作为子类型合并入此。

🏢

Handler Node

命名负责该路径的内部办公室或具体联系人，可由管理员设置是否对研究者可见。

⬡
Workflow Builder Canvas 截图 建议选一个包含 4 种节点类型的 MTA 示例 flow

AI Suggestion Panel — 五种状态

管理员可上传政策文件或输入 URL，由 Claude API 解析并生成可编辑的初始决策树草稿。面板在五种明确状态之间流转，每种状态对应不同的操作语境。

Idle 待机 Reviewing 审阅输入中 Generating 生成中 Confirming 二次确认 Applied 已应用到 Canvas
C

AI Assistant 功能迭代（第一批，由我负责）

① URL 抓取 + 多模态图片理解：AI 可抓取外部链接的文字与图片内容作为生成上下文。
② 两阶段确认流程：AI 先复述对修改意图的理解，用户确认后才执行 canvas 变更，防止误操作。
③ 结构化操作展示：每条操作以「类型标签 + 受影响节点 + AI 推理说明」格式呈现，让管理员清楚知道 AI 在做什么，以及为什么。

06  ·  User Portal — 研究者端

从"发邮件猜谁负责"到"按向导走完流程"

User Portal 消费 Admin Portal 发布的路由逻辑，始终与管理员最新发布版本同步——这是两端分离架构的核心价值。

Dashboard

展示进行中的协议请求、需要注意的事项，以及进入新请求流程的入口。设计原则是"高频信息优先"，减少导航层级。

Guided Intake（核心）

引导式向导，逐节点跟随已发布的 Workflow 决策树。研究者在每个 Decision Node 选择答案，系统沿路由路径前进，最终呈现路由结果、负责办公室与联系人、需要准备的材料清单。加入"不确定"选项处理边界情况，显式确认步骤降低误报率。

Knowledge Database

按文档类型组织的协议知识库，展示各类协议的定义、涉及办公室、处理时限参考及常见问题。Phase 1 由管理员维护，Phase 2 开放研究者侧浏览与搜索。

07  ·  设计决策与取舍

几个关键的 Scoping 决策

Approval Chain 并入 Workflow Builder

原计划作为独立模块，评审后发现它本质上是 Action Node 的延伸。合并节省了导航层级，避免管理员在两处维护重复信息。

Review & Approve 推迟到 Phase 2

日常协议操作界面需与现有邮件系统深度集成，本学期资源不足以做好。决策是：聚焦路由逻辑定义侧，明确划定边界，不做半成品。

Knowledge Base 作为不可压缩的前置条件

依赖链是 KB → Workflow Builder → Guided Intake → Review & Approve。AI 生成质量和 Intake 内容准确性都直接依赖 KB 的数据质量，它不能被跳过。

用户测试采用远程桌面模式

为让参与者使用带真实 AI 功能的原型（而非 mock），我们让参与者远程接入我们的电脑操作——避免了要求参与者在本地配置 API Key 的摩擦。
